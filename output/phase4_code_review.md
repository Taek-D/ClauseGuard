**Findings**
- [CRITICAL] Cross-org IDOR exists in report and analysis endpoints. They authorize against the caller’s own org, then read or mutate records by arbitrary `contractId` without verifying that the contract belongs to that org. A user in org A can access or affect org B’s report/analysis if they know a UUID. [reports](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/reports/index.ts#L46) [reports](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/reports/index.ts#L53) [reports](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/reports/index.ts#L90) [analysis](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/analysis/index.ts#L49) [analysis](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/analysis/index.ts#L63) [analysis](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/analysis/index.ts#L71)
- [CRITICAL] The Stripe webhook is not signature-verified. The handler only checks that `stripe-signature` and `STRIPE_WEBHOOK_SECRET` exist, then trusts the JSON body. Anyone who can reach the endpoint can forge subscription updates and change org plans/status. [subscriptions](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/subscriptions/index.ts#L49)
- [CRITICAL] Exported reports are built by concatenating unescaped contract/risk text into HTML. Contract filenames, risk titles, and descriptions can inject script into the generated report page. [reports](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/reports/index.ts#L66) [reports](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/reports/index.ts#L71) [reports](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/reports/index.ts#L78)
- [WARNING] API keys are stored in plaintext and returned once from the same value persisted in the database. A DB leak exposes all active keys immediately; store a one-way hash and only keep/display the raw token at creation time. [users](/E:/프로젝트/오케스트레이션_팀/Project2_orchauto/side-project/ClauseGuard/supabase/functions/users/index.ts#L54)
- [WARNING] Several multi-step write flows ignore DB/storage errors or have no rollback path, so partial success can leave inconsistent state. Examples: registration bootstrapping, template rule replacement, and contract upload/analysis creation.
